AIDEAL Pipeline — Architecture & Data Flow

From init to execution-verified comprehension, with the self-repairing fix loop. Each card names the CLI command, the code file, and — if an LLM writes there — the prompt. Package root: grail-agent/src/aideal/.

Static code — no API key LLM prompt stage — teal cards; author = writes/grades, audience = consumes doc Memory / artifact — written by the loop, read back into prompts ↺ = fix loop  ·  prompts live in default_prompts/aideal/
Run order — one CLI command at a time
1init→ 2profile→ 3api-surface→ 4intent→ 5api-tests→ 6readme --generate→ 7scaffold→ 8comprehension→ 9--execute ↺→ 10augment→ 11grounding→ 12organize / index→ 13catalogue→ 14puzzle
↻ steps 10–13 (augment · grounding · organize · catalogue) run after --execute — that’s when the ★verified tier exists — and re-feed the next comprehension pass.
What the code actually enforces (verified by reading the source): the only hard order is step 6 first — readme --generate writes LLM_readme.md, which every later command parses. catalogue/grounding/organize are mutually independent (each recomputes tiers from LLM_readme.md + error_log) and pay off after --execute, because the ★verified tier needs a real run. The optional class-context read path prefixes each API with its class receiver + a verified sibling’s call pattern — but it recomputes that model in memory (_build_catalogue_context); it does not read the api/<Class>.md files catalogue writes, so those files are a human view, not a comprehension input. You can run 10–13 before comprehension to eyeball structure, but it won’t feed the run and has no verified tier yet. Teal = an LLM writes there.

The relationship (why it’s a loop): prompts read memory (failures, notes, aliases are injected) → the audience produces code → execution writes outcomes to error_log.jsonl → curate stages read the log and write back into the doc / notes / aliases → the next prompt reads a better doc. Every mistake permanently improves the codebase’s LLM-readiness, not just the current run (Voyager / Reflexion lineage).

Future — where each feedback component is heading
ComponentNowNext
Fix loop Retry from the real error; infra failures stop early instead of burning rounds. Per-class receiver construction (build a sample instance from a tested sibling); feed verified successes back, not only failures.
Fix hint Spark/Hadoop patterns — Wrong FS → path.getFileSystem, output-format, Task-not-serializable. Daikon-style invariant mining: turn “it ran” into “here is the contract it obeyed” (pre/postconditions from real runs).
Error log Failures + pass/fixed, indexed by function name; carries version + timestamp. Verified-call entries indexed by type signature, so a working RasterRDD[Int] call primes a different, untried function; staleness auto-invalidation on version/age.
