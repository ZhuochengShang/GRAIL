AIDEAL — Design Flow, Code & the LLM Prompts

The logical shape of the method — intent → doc → catalogue → comprehension test → fix loop — with the code file, purpose, and (where an LLM is called) the actual prompt at each step.

This is the design flow, not the command run-order or the operational feedback plumbing. Teal = an LLM is invoked. Everything points at one test: can a model use the API from the doc alone?

4 · Memory & how the pieces relate

The design’s external, structured memory — written when tests fail/pass, read back into the prompts.

error_log.jsonl

error_log.py · ErrorLog

Every fail / pass / fixed record (+ version + timestamp). Feeds the retry (failures_for), the doc (augment), and alias mining.

notes_to_self.md

notes_to_self.py

Repeated errors distilled to compact lessons, injected into later generation prompts.

LLM_readme + catalogue

readme_agent.write_catalogue

The doc itself: entries, then index + per-class shards, tier-badged and receiver-typed.

alias map

error_log.suggest_aliases · alias_registry.py

Names the model expected (its hallucinations) → alias affordances, so a “wrong” call still resolves.

The relationship: prompts read memory (facts, failures, aliases are injected) → the audience produces code → the test writes the outcome to error_log.jsonl → curate steps read the log and improve the doc / notes / aliases → the next prompt reads a better doc. Every mistake permanently raises the codebase’s LLM-readiness, not just this run.
