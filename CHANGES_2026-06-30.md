# GRAIL / AIDEAL — session changes (2026-06-30)

A guided summary of what changed, grouped by theme. Guiding principle throughout:
**general mechanism in AIDEAL, codebase specifics in config/adapter, and the user
writes as little as possible (ideally zero code).**

## 1. Generalized, layered prompts

- `intent_common`, `readme_distill`, `readme_entry`, `comprehension_write`,
  `comprehension_write_exec` are now **codebase-agnostic** — no raster/Raptor/Scala
  hardcoding. Domain arrives only through `project_profile.yaml`.
- Editable copies live in `grail-agent/prompts/aideal/`; the package `default_prompts/`
  are kept in sync as the pip-install fallback.
- Framework specifics are injected through slots, not baked into prompts:
  - `{exec_hints}` — generic Spark "transformations are lazy; force an action or write" —
    lives in the **scala-spark adapter**.
  - `{io_hints}` — codebase load/write cheat-sheet — see §4.

## 2. Intended-API selection

- **LLM "commonly-used" signal on by default** (`intent.use_llm: true`, `llm_common: 4`);
  degrades to static scoring with no key/profile.
- New **`aideal intent-compare`** — scores the surface static-only vs static+LLM and
  reports counts, Jaccard, and which APIs each side adds (quantifies the LLM signal).
- The `intent_common` prompt now excludes framework plumbing; scope is controlled by
  `source_globs` (dropping test/web/server modules removes most noise at the source).

## 3. README generation + augment-from-log

- `readme --generate` grounds each entry on signature + distilled docs + **mined test
  usage**; distill runs once and is cached.
- **New `aideal augment`** — folds real failures/fixes from `error_log.jsonl` into each
  entry's `Common Failure Modes` and `Fix Code Hint` (evidence only, re-runnable).
- Finding: APIs **with** test grounding produce correct docs (mapPixels passed); APIs
  **without** (raptorJoin/raptorJoinFeature) hallucinated method-style calls and failed —
  a clean, quantifiable doc-quality signal.

## 4. Execution-grounded comprehension (`--execute`)

- **Single-API testing**: `aideal comprehension --api <name> --execute` isolates one API.
- **Fix loop default = 3 rounds** (was 1). Each retry sees the prior error + fix hint from
  the error log; a working fix is logged as `status: fixed` for reuse.
- **Zero-config I/O** (the user writes no Scala):
  - Sample data **auto-discovered** from a `fixtures/` folder — top-level files → typed
    bindings (`raster_tif`, `vector_geojson`, `table_csv`), subfolders → `raster_dir`,
    `vector_dir`, `table_dir`. Explicit `sample_data:` entries override per binding.
  - **`preamble: auto`** — the author model generates the typed-input load code once from
    the codebase's own documented readers (cached to `docs/preamble.scala`), so a per-API
    test writes ONLY the API call. I/O can no longer fail the test.
  - **`io_hints: auto`** — the author model derives the load/write cheat-sheet once from the
    readme's reader/writer entries (cached to `docs/io_hints.txt`).
  - Both inject into **every fix-loop round**.
- **The spark-submit command lives in the scala-spark adapter** (parameterized by
  `packages`/`repositories`); a project supplies only `jars`, `packages`, `repositories`,
  and (for Beast's `RasterRDD` collision) a curated `imports` list.
- Insight surfaced: early "failures" were a scaffold **import collision** and a **writer
  API**, not the doc — motivates tagging failures by origin (doc / harness / other-API).

## 5. Demo agent (`experiments/rdpro/demo_agent.py`)

- Small agent: **text / Python → runnable RDPro Scala**, grounded on the `LLM_readme`,
  the alias registry, and the incremental error-log/fix-guide loop.
- `--dry-run` exercises the whole loop offline; `--execute` runs it for real.

## 6. Deliverables

- `paper/eval_benchmark_brief.md` — metrics tiers, baseline conditions (C−/C_sig/C0/C1/C2
  + RAG), related benchmarks (AutoAPIEval, CoderEval, DS-1000, private-library work),
  positioning ("codebase as the unit of intervention").
- `GRAIL_AIDEAL_progress.pptx` — added Evaluation-Plan and LATEST slides.

## The self-improving loop (net effect)

```
comprehension --api X --execute   # isolates X; 3-round fix loop; logs errors/fixes
aideal augment                    # fold learnings into the readme entry
comprehension --api X --execute   # next run injects them as known_failures
```

Each pass makes the doc — and the next generation — better, with no hand-written Scala.
